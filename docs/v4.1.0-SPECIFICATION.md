## INTELLIGENT OPTIMIZATION: QUALITY-FIRST APPROACH

**Core Philosophy**: Default to QUALITY, optimize only when safe. Never make Claude "dumb" for cost savings.

### Intent Detection Layer (NEW - Critical for Safety)

Before ANY optimization, understand query intent:

**Intent Classification**:
```
QUICK ANSWER:
  Keywords: "quick", "summary", "just", "briefly", "tl;dr"
  Sentiment: Neutral/positive (user satisfied with summary)
  → Safe to cache, compress, truncate
  → Example: "Quick summary of this code"

DETAILED ANALYSIS:
  Keywords: "detailed", "explain", "why", "how", "comprehensive"
  Sentiment: Request for understanding (user wants depth)
  → UNSAFE to cache, compress, truncate
  → Return fresh Claude response with full reasoning
  → Example: "Detailed security analysis of this code"

FOLLOW-UP REFINEMENT:
  Keywords: "more", "additional", "also", "show", "what about"
  Context: Previous query was cached
  → UNSAFE to use cache from previous query
  → Return fresh analysis on new angle
  → Example: Previous "Find XSS", now "Also check CSRF"

ROUTINE/REPETITIVE:
  Keywords: Same query repeated (exact match)
  Sentiment: Neutral (routine check)
  → SAFE to cache (100% identical)
  → Return cached response
  → Example: Daily code scan, same parameters

LEARNING/EXPLORATION:
  Keywords: "What if", "Try", "Experiment", "Compare"
  Sentiment: Exploratory (user learning)
  → UNSAFE to cache (different scenarios)
  → Return fresh Claude responses
  → Example: "What if we used async instead?"

CACHE BYPASS (EXPLICIT USER CONTROL):
  Keywords: "--no-cache", "--fresh", "bypass cache", "skip cache", "!"
  Behavior: User explicitly requests fresh response, never use cache
  → FORCE skip all cache layers (Layers 1-5 skipped entirely)
  → Go directly to Claude API (Layer 6)
  → Even if exact duplicate in cache, return fresh
  → Example: "--no-cache Analyze this" or "! Find functions"
  
  Implementation:
    ├─ Prefix patterns: "--no-cache", "--fresh", "!"
    ├─ Suffix patterns: "(no cache)", "(fresh)", "(bypass)"
    ├─ Priority: Explicit user override beats any classifier decision
    └─ Audit: Log every bypass request (user wants fresh for reason)
```

**Sentiment Indicators & Bypass Patterns**:
- "Detailed", "comprehensive", "explain", "why", "how" → Need fresh response
- "Quick", "brief", "summary", "tl;dr" → Safe to cache
- "More", "also", "what about", "additionally" → Indicate refinement request
- Question marks + complex sentence → Likely needs detail
- Exclamation marks, urgency words → Request for fast summary acceptable

### Implementation: Unified Classifier Code Structure

```go
// internal/intent/classifier.go — Single classifier outputs both decisions

type IntentDecision struct {
  Intent          IntentType      // QUICK, DETAILED, ROUTINE, LEARNING, FOLLOW_UP
  CacheSafe       bool            // Should we cache this?
  RecommendModel  ModelType       // Haiku, Sonnet, Opus
  MaxTokens       int             // Output budget
  OptimizeMode    OptimizeLevel   // Aggressive, Moderate, Minimal
  Confidence      float64         // 0.0-1.0 (higher = more certain)
  FeedbackBonus   float64         // Bonus/penalty from user history
}

func (c *IntentClassifier) Classify(
  query string,
  userHistory UserFeedbackHistory,
  context PreviousQueries,
) *IntentDecision {
  
  // Step 1: Base intent from keywords/sentiment
  baseIntent := c.detectIntent(query)
  baseModel := c.intentToModel(baseIntent)      // QUICK → Haiku
  baseCacheSafe := c.isCacheSafe(baseIntent)    // QUICK → true
  
  // Step 2: Modulate with user history
  feedback := userHistory.GetFeedbackPattern()
  confidence := c.calculateConfidence(baseIntent, feedback)
  
  if feedback.PrefersModeDetail && baseIntent == QUICK {
    // User consistently wants more detail than intent suggests
    baseModel = SONNET
    baseCacheSafe = false
    confidence -= 0.2  // Lower confidence in QUICK classification
  }
  
  // Step 3: Return unified decision
  return &IntentDecision{
    Intent:         baseIntent,
    CacheSafe:      baseCacheSafe,
    RecommendModel: baseModel,
    MaxTokens:      c.maxTokensFor(baseIntent),
    OptimizeMode:   c.optimizeModeFor(baseCacheSafe),
    Confidence:     confidence,
    FeedbackBonus:  feedback.RecentAccuracy,
  }
}
```

## CONFIGURABLE OPTIMIZATION FRAMEWORK: Plugin Architecture

**Core Philosophy**: Users configure ANY optimizations they want (from internet, custom, built-in). Gateway applies them in order, with security layer always on top.

### Configuration-Driven Architecture

```yaml
# config.yaml — User configures all optimizations

gateway:
  port: 8080
  security_layer: enabled  # ALWAYS active, on all responses

optimizations:
  # 1. Configure RTK optimization (example 1)
  rtk:
    enabled: true
    config:
      command_proxy_savings: 99  # RTK saves 99.4% on command output
      models:
        low_effort: haiku        # RTK + auto-effort coupling
        medium_effort: sonnet
        high_effort: opus
      
  # 2. Configure MCP tools (example 2)
  mcp:
    enabled: true
    tools:
      - type: "web_scraping"
        name: "scrapling"
        settings:
          css_selector: true        # Always extract section only
          markdown_only: true       # Token-efficient format
          cache_responses: true
      
      - type: "code_analysis"
        name: "builtin_lsp"
        settings:
          use_lsp: true            # Structured symbol search, not grep
          cache_symbols: true
      
      - type: "database"
        name: "builtin_sql"
        settings:

## RESPONSE QUALITY IMPACT ANALYSIS

**CRITICAL**: Token optimization can degrade response quality if done wrong. Each layer has quality implications.

### Layer-by-Layer Quality Impact

**LAYER 1: Exact Deduplication** ✅ NO QUALITY IMPACT
```
Mechanism: Return cached response from identical request
Quality: Perfect (same request gets same response)
Risk: None (if cache invalidation is correct)
When to apply: Always safe
```

**LAYER 2: Security Validation** ✅ NO QUALITY IMPACT (improves it)
```
Mechanism: Block injection attacks before Claude sees them
Quality: Better (prevents nonsensical Claude responses to malicious input)
Risk: Might block legitimate edge cases (e.g., user analyzing SQL malware)
Mitigation: Whitelist mode for advanced users, audit logs

Example problem:
  User wants to analyze SQL injection: "SELECT * FROM users WHERE id = ' OR '1'='1'"
  Our validator might block this (looks like SQLi attack)
  Solution: Add context flag --allow-sql-examples
```

**LAYER 3: Knowledge Graph Queries** ✅ NO QUALITY IMPACT (limited scope)
```
Mechanism: Answer from indexed relationships instead of Claude reasoning
Quality: Perfect for graph queries, but only answers relationship questions
Risk: Can't answer "why" or "how" - graph only has "what"

Example safe:
  Q: "Find all functions calling authenticate()"
  A: Direct graph lookup, 100% accurate

Example unsafe:
  Q: "Is this secure code?"
  A: Graph can't answer (needs Claude reasoning)
  Solution: Graph hit only for relationship queries, fallback to Claude for reasoning
```

**LAYER 4: Input Optimization** ⚠️ MINOR QUALITY IMPACT (acceptable)
```
Sub-layer 4a: Tool Stripping
  Before: Claude sees 300 tools, picks 5 relevant
  After: Only send 5 relevant tools
  Quality impact: Minimal (Claude has fewer distractions)
  Risk: Occasionally misses useful tool
  Mitigation: Always include "general" tools (web search, code analysis)

Sub-layer 4b: Parameter Compression
  Before: {user_id: 123, include_metadata: true, max_results: 100}
  After: {u: 123, m: 1, n: 100}
  Quality impact: None (compressed params still parse correctly)
  Risk: None

  Before: "Find Python functions validating user input"

  Quality impact: None (same intent, clearer structure)
  Risk: None
```

**LAYER 5: Semantic Cache** ⚠️ POTENTIAL QUALITY IMPACT (needs thresholding)
```
Mechanism: Return cached response for SIMILAR (not identical) request
Quality impact: Depends on similarity threshold

Threshold: cosine >0.85 (strict, conservative)
  Q1: "Find all functions calling authenticate()"
  Q2: "List all functions that call authenticate()"
  Similarity: 0.92 (high, almost identical intent)
  Response: Use cached answer from Q1
  Quality: Excellent (semantically identical)

Threshold: cosine >0.70 (loose, aggressive)
  Q1: "Find authentication bugs"
  Q2: "Find security issues"
  Similarity: 0.72 (moderate, related but different)
  Response: Use cached answer from "Find auth bugs" for "Find security"
  Quality: POOR (wrong answers for wrong questions) ❌

Recommendation: Use >0.85 threshold
  - 98% accurate matching
  - False positive rate <0.1% acceptable
  - Conservative side of trade-off

Quality assurance:
  - Monitor false positive rate (target: <0.1%)
  - A/B test: return cached vs fresh, compare quality
  - User feedback: "Was this answer correct?" button
  - Kill criterion: false positives >0.5% (pause semantic cache)
```

**LAYER 6: Claude API Call** ✅ NO QUALITY IMPACT (uses native Anthropic caching)
```
Mechanism: Anthropic's prompt cache handles system prompt caching
Quality: No impact (same system prompt, same Claude behavior)
Risk: None
```

**LAYER 7: Output Optimization** ⚠️ POTENTIAL QUALITY IMPACT (context-dependent)

Sub-layer 7a: Sanitization
  Quality impact: None (removes malicious content, preserves legitimate response)
  Risk: Might over-sanitize (block `<` in code examples)
  Solution: Context-aware sanitization (know if output is code vs prose)

Sub-layer 7b: Response Compression
  Before: "The function authenticate() located in auth.go at line 45..."
  After: {file: "auth.go", line: 45, ...}
  Quality impact: Depends on format
  Risk: Loses Claude's explanation/reasoning
  Solution: Only compress for structured queries, keep prose for reasoning

Sub-layer 7c: Field Filtering
  Before: Full response with reasoning
  After: Only {id, name, file} (requested fields)
  Quality impact: Acceptable if user requested summarization
  Risk: Loses context user didn't ask for
  Solution: Always include "summary" field, let user request details

Sub-layer 7d: Delta Output
  Before: Full response (500 tokens)
  Same code, same query: "No changes" (0 tokens)
  Quality impact: EXCELLENT (no new bugs = no changes needed)
  Risk: User misses new issue if code DID change
  Solution: Compare hashes, only use delta if code truly unchanged

  Risk: High (user misses key finding)
